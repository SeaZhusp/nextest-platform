# 测试智能体平台 — 完整演进方案

本文档描述测试用例生成智能体从 MVP 到生产可用的**分阶段目标、详细需求、验收标准与非功能要求**，并与后端分层架构（`api → services → repositories → models`）对齐。

**文档版本**：1.1  
**适用仓库**：`nextest-platform`（前端 + 后端）

**修订说明（1.1）**：阶段一「用户输入」补充 **多模态扩展预留**（MVP 仍仅实现文本）。

---

## 1. 总体目标

### 1.1 产品愿景

构建一个以 **LLM + 可插拔技能（Skill）** 为核心的测试智能体能力：支持从自然语言需求生成结构化测试用例，支持流式输出与人工编辑；逐步增强 **知识库（RAG）**、**多轮对话与记忆**、**历史用例引用与版本管理**，最终达到可部署、可观测、可权限隔离的生产形态。

### 1.2 架构原则

| 原则 | 说明 |
|------|------|
| 分层清晰 | 路由层轻量；业务在 `services`；数据访问在 `repositories`；ORM 在 `models`。 |
| 技能可插拔 | 技能包位于 `backend/skills/<skill_id>/`，由注册表扫描/加载；业务代码禁止写死技能实现路径。 |
| 执行单一入口 | 技能调用统一经 `services/skill/executor`（或等价模块），便于鉴权、审计、限流与错误处理。 |
| 记忆以 DB 为准 | 会话与消息持久化在数据库；Redis 仅作可选缓存/加速，不作为唯一真相源。 |
| 流式契约稳定 | SSE（或 WebSocket）事件类型枚举清晰，前后端版本可独立演进。 |
| 输入可演进 | 用户消息采用 **统一「内容片段」模型**（如 `parts[]`：`text` / 未来 `image` / `audio` 等）；**一期仅实现 `text`**，其余类型在 API、DB、LLM 适配层 **预留但不实现**。 |

### 1.3 核心后端模块（演进式启用）

| 模块 | 职责 | 阶段 |
|------|------|------|
| `api/endpoints/agent` | 对话/生成主入口（流式） | 一 |
| `api/endpoints/skills` | 技能列表、元数据（可选：受控调试执行） | 一 |
| `api/endpoints/testcases` | 用例 CRUD、版本相关接口 | 一～三 |
| `api/endpoints/documents` | 文档上传与知识库管理 | 二 |
| `api/endpoints/traces` | 轨迹/审计查询 | 三起强化 |
| `services/llm` | 统一 LLM 客户端、流式、成本记录 | 一 |
| `services/skill` | `registry` / `executor` / `base` | 一 |
| `services/agent` | `engine`（ReAct 或简化循环）、`context`（上下文组装）、`prompts`（引擎级提示） | 一 |
| `services/rag` | 分块、检索、嵌入、向量存储抽象 | 二 |
| `services/task` | 异步任务、进度（Celery 或等价） | 四（批量场景） |
| `services/export` | Excel / Markdown 导出 | 四 |
| `backend/skills/*` | 具体技能包（如 `test_case_gen`） | 一 |

## 2. 阶段一：MVP — 动态 Skill + 会话记忆 + 流式生成 + 用例落库

**周期建议**：2～3 周（含动态技能与持久化记忆，较原「纯单轮 MVP」工作量增加）

### 2.1 阶段目标

跑通 **「输入需求 → 选择/默认 Skill → LLM 流式生成结构化用例 → 前端展示与编辑 → 保存数据库」** 闭环；同时实现 **技能动态注册与执行**、**会话级记忆**（多轮对话与刷新可恢复），为后续 RAG 与复杂 ReAct 打基础。

### 2.2 功能需求

#### 2.2.1 用户输入

**MVP 范围（一期必达）**

| 编号 | 需求描述 | 验收标准 |
|------|----------|----------|
| F1.1 | 提供 **文本** 输入能力，用于描述测试需求。 | 请求可提交文本；服务端校验 **≤ 5000** 字符（可配置）。实现方式二选一且需在 `schemas` 中固定：**(A)** 顶层字段 `content: string`；或 **(B)** `parts: [{ "type": "text", "text": "..." }]` **仅允许单段 text**（推荐 B，与下表多模态预留一致）。 |
| F1.2 | 支持在「同一会话」内多轮补充需求。 | 每次请求携带 `session_id`；无则创建新会话并返回新 `session_id`。 |

**多模态扩展（一期预留口子，不实现非文本能力）**

产品规划上用户输入可能演进为 **多模态**（如图片、语音、短附件）。一期 **不**交付图片理解、语音转写等能力，但须在 **契约、数据与调用链** 上预留，避免后期大改表结构或推翻 API。

| 编号 | 需求描述 | 验收标准（预留） |
|------|----------|------------------|
| F1.19 | **API 契约**：消息内容为「片段列表」模型。 | `schemas/agent.py`（或等价）定义 `MessagePart`：`type` 枚举至少包含 `text`；预留 **`image_url` / `image_base64` / `audio_url` / `file_ref`** 等（具体名称以代码为准，一期 **禁止**使用或收到即返回明确「暂不支持」）。请求体验证：一期仅允许 `type=text` 的 part（或唯一 `content` 字符串映射为单段 text）。 |
| F1.20 | **持久化**：`messages` 存完整结构。 | 用户消息建议存 **`content_json` / `parts` JSON**（或 ORM JSON 字段），而非仅 `TEXT` 一列；一期写入仅含 text 片段。历史消息反序列化后可直接交给上下文组装。 |
| F1.21 | **上下文组装**：`services/agent/context.py`（或等价）按片段类型组装。 | 一期逻辑：仅处理 `text` → 拼进 prompt。对未知 `type` 预留分支（记录日志或跳过），便于二期接多模态。 |
| F1.22 | **LLM 适配层**：`services/llm` 统一入口支持「多段用户内容」。 | 一期实现：**仅 text** → 映射为供应商 chat 的 `user` 字符串或单条 user message。预留将 `image_*` 转为 vision API 的 `content` 数组（方法占位或接口即可，无需真实调用）。 |
| F1.23 | **附件与对象存储（可选预留）**。 | 若采用 `file_ref`：预留 **attachment_id + 存储桶路径** 字段或独立 `attachments` 表（一期可为空表或不创建，仅文档约定）；**二期可与文档上传能力复用**同一存储策略。 |
| F1.24 | **Skill 入参**：技能执行上下文携带「标准化消息」。 | `executor` / `BaseSkill` 传入的 user 输入为 **与 API 一致的 parts 或封装对象**；`test_case_gen` 一期只读 text 段；后续可同上下文增加图片等而无改签名。 |

**后期阶段（写入计划，非一期验收）**

| 阶段 | 多模态相关交付（建议） |
|------|------------------------|
| 二 | 与「文档/文件」能力协同：图片/截图可走 **上传 → 存储 → file_ref**；是否调用 vision 模型由配置开关控制。 |
| 三 | 会话内引用「带图消息」、参考示例含截图时，上下文与轨迹中展示附件元数据。 |
| 四 | 权限：附件仅同用户/同项目可见；审计与配额（大小、次数）。 |

#### 2.2.2 动态 Skill 加载与执行

| 编号 | 需求描述 | 验收标准 |
|------|----------|----------|
| F1.3 | 技能包独立于 `app` 代码树，按目录加载。 | 存在 `backend/skills/<skill_id>/`（含 `skill.py`、`config.json` 等约定文件）；启动时 **注册表** 能列出已发现技能。 |
| F1.4 | 运行时按 `skill_id` 解析并执行，禁止散落硬编码 import。 | 所有执行经 **统一 executor**；非法 `skill_id` 返回明确错误码/文案。 |
| F1.5 | 首期内置「测试用例生成」技能。 | 默认 `skill_id`（如 `test_case_gen`）可生成结构化测试用例列表（见 F1.6）。 |
| F1.6 | 生成结果结构固定。 | LLM 输出经解析后为 **JSON 数组**；每项至少包含：`id`/`case_no`、`module`、`title`、`preconditions`、`steps`、`expected`、`priority`（字段名以 `schemas` 为准）。 |
| F1.7 | 至少生成条数。 | 在需求足够前提下 **≥ 3 条**用例；不足时由提示词约束并在空场景下友好提示。 |

#### 2.2.3 LLM 与流式输出

| 编号 | 需求描述 | 验收标准 |
|------|----------|----------|
| F1.8 | 调用外部 LLM API（如 DeepSeek 等），配置可切换。 | `core/config` 与 `.env.example` 包含模型相关项；密钥不入库。 |
| F1.9 | 后端 **SSE** 向客户端推送生成过程。 | 客户端在完整响应前即可收到流式事件；协议文档化（如 `token` / `message_delta` / `done` / `error`）。 |
| F1.10 | 非 JSON 流式场景。 | 若采用「先流式文本再整体解析 JSON」，需在流结束后做一次解析；失败走 **F1.18**。 |

#### 2.2.4 会话与记忆

| 编号 | 需求描述 | 验收标准 |
|------|----------|----------|
| F1.11 | 持久化会话与消息。 | 表或等价结构：`sessions`、`messages`；每条用户消息与助手最终回复均可查询。 |
| F1.12 | 上下文窗口策略。 | 每次调用 LLM 前组装 **最近 K 轮**（建议 **K=6**，可配置）用户+助手消息进入 prompt；与 `services/agent/context.py` 行为一致。 |
| F1.13 | 刷新或重新进入后记忆不丢。 | 同一 `session_id` 再次请求时，历史对话仍参与上下文（在 K 轮限制内）。 |

#### 2.2.5 用例展示与保存（前后端协同）

| 编号 | 需求描述 | 验收标准 |
|------|----------|----------|
| F1.14 | 前端将 JSON 用例渲染为可编辑表格（类 Excel）。 | 单元格可编辑；具体 UI 实现由前端完成，后端提供与 schema 一致的 JSON。 |
| F1.15 | 用户修改后点击保存，写入数据库。 | `testcases` 相关 API 可用；保存成功后刷新页面数据仍存在。 |
| F1.16 | 初期数据库。 | 开发环境 **SQLite** 可接受；连接串可配置，便于后续换 PostgreSQL。 |

#### 2.2.6 错误处理与重试

| 编号 | 需求描述 | 验收标准 |
|------|----------|----------|
| F1.17 | 网络或 LLM 失败。 | 统一错误结构；前端可提示并可 **重试**（不白屏、不崩溃）。 |
| F1.18 | LLM 返回非合法 JSON。 | 明确错误提示「解析失败」；可建议用户缩短需求或重试；服务端记录日志。 |

### 2.3 非功能要求（阶段一）

| 编号 | 类型 | 要求 |
|------|------|------|
| N1.1 | 响应时间 | 单次生成（不含 LLM 供应商排队）目标 **≤ 15 秒**；超时可配置并返回友好错误。 |
| N1.2 | 部署 | 开发环境 **localhost** 可完整运行（前端 + 后端 + DB）。 |
| N1.3 | 安全基线 | Skill 仅从白名单目录加载；executor 层可记录调用者与 `skill_id`（为四期审计铺路）。 |

### 2.4 交付物与接口约定（阶段一）

- **API**：流式对话/生成（如 `POST /api/.../agent/chat` 或项目既有前缀下）；用例 CRUD；可选 `GET /api/.../skills` 列出技能；用户消息体 **符合 F1.1 / F1.19**（文本必达，多模态字段预留）。
- **数据**：`sessions`、`messages`（建议 **JSON 字段承载 parts**，满足 F1.20）、`testcases`（最小字段满足 F1.6/F1.15）。
- **技能包**：至少 `backend/skills/test_case_gen/`（入参符合 F1.24）。
- **文档**：SSE 事件类型说明、环境变量说明（README 或本文档附录引用）；**MessagePart / 多模态预留** 在 `schemas` 中有注释或与本文 **§6.5** 一致。

## 3. 阶段二：文档上传 + RAG 知识库

**周期建议**：约 2 周

### 3.1 阶段目标

使智能体能基于用户上传的 **需求文档、历史说明** 等进行检索增强生成；支持 **文档生命周期管理** 与 **按项目隔离的知识库**。

### 3.2 功能需求

#### 3.2.1 文档上传

| 编号 | 需求描述 | 验收标准 |
|------|----------|----------|
| F2.1 | 前端支持拖拽或点击上传。 | 交互可用；显示文件名与上传进度/结果。 |
| F2.2 | 支持格式与大小。 | **pdf、docx、md、txt**；单文件 **≤ 20MB**（可配置）。 |
| F2.3 | 上传后状态可见。 | 展示解析/索引状态（排队中、处理中、完成、失败及原因）。 |

#### 3.2.2 解析与分块

| 编号 | 需求描述 | 验收标准 |
|------|----------|----------|
| F2.4 | 解析为纯文本。 | 各格式能抽取正文；异常时失败状态可追踪。 |
| F2.5 | 分块策略。 | 按语义分块；参数目标 **约 500 tokens/块，重叠 50**（具体以 tokenizer 或字符近似实现，需在配置中写明）。 |
| F2.6 | 可观测性。 | 用户或管理端可查看 **块数量** 等元数据。 |

#### 3.2.3 向量化与存储

| 编号 | 需求描述 | 验收标准 |
|------|----------|----------|
| F2.7 | 嵌入模型。 | 如 `text-embedding-3-small` 或等价；API Key 经配置注入。 |
| F2.8 | 向量数据库（初期）。 | **Chroma** 本地或可配置实例；抽象层便于三期/四期切换 Qdrant/Milvus。 |
| F2.9 | 写入与关联。 | 向量与 `document_id`、`project_id` 关联，便于删除与隔离。 |

#### 3.2.4 检索增强生成（RAG）

| 编号 | 需求描述 | 验收标准 |
|------|----------|----------|
| F2.10 | 检索条数。 | 每次用户提问检索 **Top 3～5** 相关文本块（可配置）。 |
| F2.11 | 拼入提示词。 | 检索结果进入 **系统提示或独立 system 片段**，再调用阶段一已有生成链路。 |
| F2.12 | 效果可感知。 | 验收场景下生成内容能体现文档中的 **术语/规则**（可用固定样例文档演示）。 |

#### 3.2.5 知识库管理

| 编号 | 需求描述 | 验收标准 |
|------|----------|----------|
| F2.13 | 删除文档。 | 用户可删除已上传文档；**元数据与向量同步删除**（或标记不可检索）。 |
| F2.14 | 多项目隔离。 | 不同 `project_id` 检索仅命中本项目知识库；无串库。 |

### 3.3 非功能要求（阶段二）

| 编号 | 类型 | 要求 |
|------|------|------|
| N2.1 | 处理耗时 | 文档解析 + 向量化 **≤ 30 秒**（视文档大小，大文件可分片异步，界面显示进度）。 |
| N2.2 | 检索延迟 | 单次检索 **≤ 1 秒**（同机/局域网 Chroma 预期可达成；需监控）。 |

### 3.4 依赖与架构补充

- 新增/扩展：`services/rag`（`chunking`、`embedding`、`retriever`、`vector_store`）、`api/endpoints/documents`、`models/repositories` 中文档与索引状态。
- 异步：大文件可走后台任务（初期可用 FastAPI `BackgroundTasks` + 轮询；四期再与统一任务框架对齐）。

## 4. 阶段三：多轮深度协作 + 历史用例引用 + 版本对比

**周期建议**：约 2 周

### 4.1 阶段目标

在阶段一记忆能力之上，强化 **产品化会话管理**、**自然语言修改整套用例**、**历史用例检索与「参考示例」**、**修改前后版本对比与确认保存**；完善 **Agent 轨迹** 便于排障。

### 4.2 功能需求

#### 4.2.1 会话管理

| 编号 | 需求描述 | 验收标准 |
|------|----------|----------|
| F3.1 | 会话列表与切换。 | 用户可创建、查看、切换不同「生成会话」。 |
| F3.2 | 完整历史展示。 | 进入某会话可看到 **全部** 历史问答与生成结果（展示策略可与性能平衡：如分页加载）。 |

#### 4.2.2 多轮修改与智能体能力

| 编号 | 需求描述 | 验收标准 |
|------|----------|----------|
| F3.3 | 自然语言修改已生成用例。 | 示例：「把优先级全部改为高」「增加一条异常场景」——助手返回 **修改后的完整用例集**（结构仍符合 schema）。 |
| F3.4 | 引擎与技能协同。 | `services/agent/engine` 支持 **ReAct 或等价的工具调用循环**；工具包含对 `test_case_gen` 或「用例重写」能力的调用；轨迹可记录。 |

#### 4.2.3 短期记忆（与阶段一衔接）

| 编号 | 需求描述 | 验收标准 |
|------|----------|----------|
| F3.5 | 最近 6 轮对话。 | 每次请求携带或服务端拉取最近 **6 轮** 用户+助手消息；指代消解可用（如「上面那条用例」在验收场景可演示）。 |

*说明：阶段一若已实现 K=6，本阶段侧重 **产品化与智能体修改链路**；若阶段一为更小 K，本阶段对齐为 6。*

#### 4.2.4 历史用例检索与参考示例

| 编号 | 需求描述 | 验收标准 |
|------|----------|----------|
| F3.6 | 检索当前项目下历史用例。 | 支持 **关键词** 检索；可选 **向量** 检索（与用例 embedding 策略一致）。 |
| F3.7 | 选择参考示例。 | 用户可选某条历史用例作为 **参考**；生成风格/结构与参考 **一致倾向**（验收用例固定演示）。 |

#### 4.2.5 用例版本与对比

| 编号 | 需求描述 | 验收标准 |
|------|----------|----------|
| F3.8 | 版本链。 | 用例存在 **版本** 或 `parent_id` 链；保存新版本不覆盖旧版本（或保留可追溯关系）。 |
| F3.9 | Diff 展示。 | 修改前后 **简单 diff**（字段级或文本级均可，需产品确认粒度）；用户 **确认** 后落库为新版本。 |

#### 4.2.6 轨迹与审计

| 编号 | 需求描述 | 验收标准 |
|------|----------|----------|
| F3.10 | Agent 轨迹查询。 | `traces` 或等价：可按 `session_id`、时间查询工具调用、错误、耗时（摘要级即可）。 |

### 4.3 非功能要求（阶段三）

| 编号 | 类型 | 要求 |
|------|------|------|
| N3.1 | 历史用例检索 | 平均 **≤ 500ms**（数据量较小时；需索引与分页）。 |
| N3.2 | Redis（可选） | 若引入 Redis：用于缓存会话列表、热点上下文等；**DB 仍为权威**；与阶段一不冲突。 |

### 4.4 数据与模型补充

- `agent_traces`（或等价）结构化存储。
- `testcases` 扩展版本字段或与 `testcase_versions` 关联表。

## 6. 跨阶段统一约定

### 6.1 API 与版本

- HTTP API 统一挂在项目约定的前缀（如 `/api`）；破坏性变更通过版本化迁移或前后端协同发布处理。

### 6.2 Prompt 分层

| 层级 | 位置 | 内容 |
|------|------|------|
| 引擎级 | `services/agent/prompts.py` | 安全策略、ReAct/工具格式、全局行为。 |
| 技能级 | `backend/skills/<id>/prompts.py` | 领域提示、输出 schema、示例。 |
| 运营级（可选） | DB `prompt_template` | 覆盖与灰度；需版本号与回滚策略。 |

### 6.3 流式与持久化时机

- **用户消息**：进入会话即可持久化（或收到请求即写）。
- **助手消息**：建议在 **流式结束** 后写入最终全文，避免半句入库；若需断线续传，可另设「草稿」状态（可选）。

### 6.4 错误码与体验

- 所有错误经统一异常与 HTTP 体；前端可映射为可重试/不可重试提示。

### 6.5 用户输入与多模态（契约摘要）

- **一期**：仅 **文本**；长度限制见 F1.1；`parts` 仅单段 `text` 或等价 `content` 字符串。
- **结构**：优先使用 **`parts[]`**（`MessagePart`），与 OpenAI/Anthropic 多模态 message content 数组概念对齐，降低后续适配成本。
- **禁止**：一期若客户端发送非 `text` 的 part，返回 **400** 与明确文案「当前仅支持文本」，或按 F1.19 定义处理。
- **持久化**：`messages` 保存 **完整片段 JSON**，便于三期展示「带图历史」时无需迁移。
- **LLM**：vision / audio 的实际上线在 **二期及以后** 分任务启用，与 `services/llm` 扩展同步。

## 8. 风险与依赖

| 风险 | 缓解 |

## 10. 附录：用户消息「片段」模型（多模态预留示例）

以下仅为 **契约示意**（非强制字段名），实现以仓库内 `schemas` 为准。

```json
{
  "session_id": "…",
  "parts": [
    { "type": "text", "text": "请根据需求生成用例……" }
  ]
}
```

后期扩展示意（**一期不验收**）：

```json
{
  "session_id": "…",
  "parts": [
    { "type": "text", "text": "请结合下图生成功能用例" },
    { "type": "image_url", "url": "https://…", "mime_type": "image/png" }
  ]
}
```

一期后端对 `image_url` 等类型应 **拒绝或忽略并文档化**，避免静默丢需求。
